Generative Model Benchmark
RealNVP vs. DDPM

Benchmarking two alternative generative models against the same conditional transition density that the training-free diffusion flow map in jump1d/ learns — on exactly the same ground-truth data, with nothing in the original pipeline modified.

The Question

Can standard generative models learn the same flow map?

p(Xt+Δt | Xt)

This is the conditional transition density of a bounded jump-diffusion process (absorbing boundaries) — the same quantity jump1d/'s training-free flow map is built to approximate. The benchmark asks: how do a normalizing flow and a diffusion model compare, trained the ordinary way, on the same task?

  • Same data: artifacts_bd/data_pairs.npz — bounded domain [0, 6]
  • jump1d/ core (common.py, data_generation.py, training.py, inference.py) is untouched
  • Pipeline is dimension-agnostic — same code could benchmark a future higher-D or unbounded SDE
Contenders

Two architectures, two very different objectives

Normalizing Flow

Conditional RealNVP

8 conditional affine coupling layers + ActNorm + permutations. Trains by maximizing exact log-likelihood via change-of-variables — the only model here that yields a closed-form NLL.

Diffusion

Conditional DDPM

UNet-style MLP denoiser with a cosine noise schedule, trained with epsilon-prediction MSE. Sampling requires iterating the full reverse chain — no closed-form density.

Evaluation

Five ways to ask "is the sampled distribution right?"

  • Wasserstein-1 distance — earth-mover's distance to ground truth
  • KL divergence — histogram-based
  • Hellinger distance — bounded density-overlap metric
  • MMD — RBF-kernel maximum mean discrepancy, unbiased estimator
  • Sampling speed, training time, peak GPU memory — practical cost
  • NLL — exact, RealNVP only (diffusion models have no closed-form likelihood)

All computed on a held-out validation split, one ML draw paired per validation Xt, so the conditioning marginal matches exactly.

Hardware

Every run — GPU, auto-detected, with mixed precision

Both models trained and sampled on the same GPU (CUDA) with the same mixed-precision settings — hardware is not a confound in this benchmark, only the architecture differs.

# jump1d/comparison/utils.py def get_device(cfg: Config) -> torch.device: if cfg.device == "auto": return torch.device("cuda" if torch.cuda.is_available() else "cpu") return torch.device(cfg.device) def amp_enabled(cfg: Config, device: torch.device) -> bool: return bool(cfg.amp) and device.type == "cuda" # AMP only engages on CUDA class GpuMemoryTracker: """Resets CUDA's peak-memory counter on entry, reads it back on exit.""" def __enter__(self): if self.device.type == "cuda": torch.cuda.reset_peak_memory_stats(self.device) return self def __exit__(self, *exc): if self.device.type == "cuda": self.peak_bytes = torch.cuda.max_memory_allocated(self.device)
  • Config.device = "auto" — picks CUDA if available, falling back to CPU only when no GPU is present.
  • Config.amp = True — torch.cuda.amp mixed precision is enabled by default, but only takes effect when the device is CUDA (no-op on CPU).
  • GpuMemoryTracker is exactly how the 37.3 MB (RealNVP) and 180.4 MB (DDPM) peak-memory figures in the results table and Figure 08 were measured — torch.cuda.reset_peak_memory_stats() at the start of training, torch.cuda.max_memory_allocated() at the end.
  • Because both models ran on identical hardware and precision, the ~3,800× sampling-speed gap (Figure 07) is attributable entirely to architecture — RealNVP's single forward pass vs. DDPM's iterative reverse chain — not to any difference in how GPU resources were used.

Conclusion

Speed vs. fidelity — and metrics disagree

  • RealNVP is dramatically cheaper at inference — ~3,800× faster sampling, ~5× less GPU memory — and wins on Wasserstein distance and exact NLL.
  • DDPM wins decisively on KL divergence and Hellinger distance (bulk-of-distribution fit), at the cost of an iterative sampler ~3,800× slower.
  • Both models trained and sampled on the same GPU with the same mixed-precision settings (device="auto", amp=True) — the speed and memory gaps above are architecture effects, not a hardware artifact.
  • The inverted ranking (RealNVP best on Wasserstein, worst on KL/Hellinger, and vice versa for DDPM) shows no single metric tells the whole story — Wasserstein is sensitive to outliers/tails, KL and Hellinger to bulk overlap.
  • The benchmark harness is model-agnostic: adding a GAN, VAE, or Flow-Matching model only requires a new models/your_model.py + train_your_model.py.

Learning Diffusion SDE Flow Maps
a step-by-step reproduction

An educational, 12-script implementation of training-free conditional diffusion models for learning stochastic flow maps in bounded domains — reproducing Figure 2 of "Generative AI Models for Learning Flow Maps of Stochastic Dynamical Systems in Bounded Domains."

The Target

Reproduce Figure 2: four histograms, one question

How much does training-data curation matter when learning a stochastic flow map near an absorbing boundary?

Step 01

1Simulate the ground truth

dXt = dWt,  X0 = 1,  0 < Xt < 6  (absorbing boundaries)

Pure Brownian motion, zero drift, unit diffusion — no parameters to tune, yet the absorbing walls make the terminal distribution non-Gaussian. Euler-Maruyama is exact for additive noise: Xn+1 = Xn + √dt · Zn. 200,000 particles, dt = 5×10-4, T = 3 → 6,000 steps.

2Build the training pairs, then noise them

Step 02 collects (xt, ΔX) pairs where both endpoints stay inside the domain. Step 03 defines the VP-SDE forward noising schedule used to turn each target increment into a denoising-training target:

Step 04

4KNN score estimation

Rather than training a score network, the score of the noised distribution is approximated directly with a k-nearest-neighbours estimator:

s(z) ≈ (μKNN − z) / varKNN

Step 05

5Reverse the noise with a probability-flow ODE

dZ/dt = −½ &beta(t)[Z + s(Z, t)]

Integrating this ODE backward turns pure noise into a recovered increment ΔX — the deterministic counterpart of the reverse diffusion process.

Steps 06–07

6Reparameterise, then train FlowNet

z = (ΔX − μ(x)) / σ(x)   —   ℒ = E[(G(x,z) − ΔX)²]

Each (x, ΔX) pair is converted into a (x, z, ΔX) label triple, then a small FlowNet (~200k parameters) is trained with plain MSE to map G(x, z) → ΔX.

Step 08

8Rollout — using the trained network as a simulator

xnew = x + G(x, z),  z ~ N(0,1) sampled fresh every step

Fresh independent noise each step replicates the independence of Brownian increments. A boundary check after each step absorbs particles that exit (0, 6), exactly mirroring ground-truth simulation.

Steps 09–11

9Why training-data curation is everything

All three conditions share the same architecture, training procedure, and rollout code. The only difference is which (x, ΔX) pairs are used to train:

  • Correct Our Method — both endpoints inside → learns the boundary-correction, matches ground truth
  • Wrong All Trajectories — includes crossings → learns the unconditional N(0,dt) step, ignores the boundary entirely
  • Wrong Only Confined — only long-survivors → effectively samples the Doob h-transform, over-suppresses movement toward walls

Conclusion

"A generative model is only as good as its training distribution"

  • The network is a universal approximator — it faithfully learns whatever distribution of transitions it's shown.
  • Clean, boundary-respecting transitions → correct boundary behaviour reproduced.
  • Naively including all transitions → boundary effect washed out entirely.
  • Filtering to only long-survivors → introduces a systematic bias away from boundaries.
  • Data curation is as important as model architecture — the central lesson of this reproduction.
